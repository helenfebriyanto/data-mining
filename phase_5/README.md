# Phase 5 Dashboard — PaySim Banking Discovery

This folder contains a Python Dash dashboard for the Phase 5 Knowledge Presentation deliverable. The dashboard is written in business English and is designed for a non-technical audience.

## Dashboard tour

1. **Executive Summary** — portfolio-level KPIs (transactions analyzed, segments, patterns, priority-queue share) plus a data-driven risk status, with three highlight cards teasing the headline findings from the other tabs.
2. **Customer Segments** — the four K-Means segments explained in plain business language, with an interactive spotlight selector, a size/risk bubble chart, and a callout on the standout finding (a 0.03%-sized segment with zero fraud labels but a 79% high-risk rate).
3. **Hidden Patterns** — association rules rewritten as "when this is seen, it usually comes with that" cards with a Coverage / Hit rate / Uplift legend in plain English, filterable by pattern type, plus the ranked uplift chart.
4. **Unusual Activities** — how the 0–6 risk score is built from four independent checks, translated fraud-validation metrics (precision/recall reframed as "how much should we trust the queue"), and the anomaly-type/method-overlap charts.
5. **Recommendations** — direct answer to "what did we discover that wasn't obvious from the raw data," followed by concrete finding → why it matters → recommended action cards.
6. **Data Explorer** — filterable, paginated table over the top 20,000 flagged transactions, for anyone who wants to look at the underlying records instead of the summaries.

## Why K-Means is used as the main segmentation reference

K-Means is used as the main customer segmentation lens because it creates four stable and interpretable segments. Direct HDBSCAN is useful as a comparison but labels too many rows as noise, while BIRCH + HDBSCAN is more useful as an outlier cross-reference for Phase 4 than as the main business segmentation layer.

## Performance design

The dashboard targets sub-100ms interactions with two complementary techniques:

1. **Tabs, the segment spotlight, and the pattern filter are fully clientside.** Every tab panel, every segment card, and every rule card is rendered once, up front, when the app starts — not rebuilt on each click. Switching between them is a single trivial **clientside callback** (pure JavaScript, defined in `callbacks.py`) that flips a `data-*` attribute; `assets/style.css` does the actual showing/hiding via CSS attribute selectors. None of this touches the Dash server, so it's effectively instant (well under 100ms) regardless of server load or network latency.
2. **The Data Explorer is the one tab that legitimately needs the server** (real filtering/sorting/paging over transaction-level rows). That table is capped at the top 20,000 flagged transactions and lives in memory after the app starts, so every filter/sort/page operation is a small, vectorized pandas operation on at most 20,000 rows — low single-digit milliseconds of actual computation. The only added latency there is the Dash round trip itself, and only that one row of interaction pays it.

Supporting choices that keep the underlying data small to begin with:

3. Use parquet and CSV outputs generated by Phase 1-4.
4. Convert them into small dashboard cache parquet files (all under a few hundred KB; the biggest, `top_suspicious_light.parquet`, is at most 20,000 rows).
5. Load those cache files exactly once when the app starts — never inside a callback.
6. Render every chart from precomputed summary tables (a handful to a few dozen rows each), never from the raw 6.3M-row dataset.

Actual latency still depends on your machine and browser, but the architecture itself never does full-table work during an interaction.

## Installation

```bash
cd phase_5_dashboard
pip install -r requirements.txt
```

## Build dashboard cache

Run this after Phase 1-4 have been executed and their parquet/CSV outputs exist.

```bash
python build_dashboard_cache.py --project-root ..
```

The script writes compact files into:

```text
phase_5_dashboard/cache/
```

## Run dashboard

```bash
python app.py
```

Then open:

```text
http://127.0.0.1:8050
```

## Fallback mode

If no dashboard cache exists, the app still opens using summary values extracted from the executed notebook outputs embedded in the submitted project archive. For final submission, build the cache from the local parquet outputs so the dashboard reflects the latest run.

## Business wording choices

- `High risk` is described as a review-priority queue, not a final fraud prediction.
- Association rules use plain-English labels: `coverage`, `hit rate inside the group`, and `uplift vs normal`.
- Customer segmentation is described as customer-transaction behavior segmentation because the unique account IDs were removed during preprocessing.
- Anomalies are framed as operational categories: possible data quality issue, rare legitimate transaction, potential risk/monitor, or potential fraud.
- Precision/recall are translated into plain sentences (e.g. "of every confirmed fraud case, this share shows up in the priority queue") before the raw metric names appear anywhere.
- The Data Explorer's `amount` column is explicitly labeled as a standardized value (0 = typical, higher = larger than typical), since dollar amounts are not recoverable after the Phase 1 scaling step — showing the raw scaled number without that caption would be misleading to a non-technical reader.
- Segment names are shown consistently as `Segment N — Name` everywhere in the app (Customer Segments tab and Data Explorer alike), even though the underlying cache files spell it two slightly different ways internally.
